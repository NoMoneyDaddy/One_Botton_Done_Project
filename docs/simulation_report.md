# 模擬使用流程與深度檢視報告

本報告基於一個**完全無代碼經驗的用戶**，在**純手機瀏覽器**環境下，嘗試開發一個「AI 圖片生成平台」的端到端模擬。

## 1. 模擬流程記錄

### 步驟 1：準備工作 (GitHub 倉庫建立)
- **預期**：用戶在 GitHub 上建立新倉庫。
- **實際體驗**：手機版 GitHub 介面建立倉庫還算順利。
- **痛點 1**：用戶可能不知道需要勾選「Add a README file」，導致後續 Codespaces 無法直接啟動。

### 步驟 2：啟動 Codespaces
- **預期**：用戶點擊 `<> Code` -> `Codespaces` 啟動雲端環境。
- **實際體驗**：在手機上，GitHub 預設會嘗試打開 GitHub Mobile App。如果用戶沒有安裝 App，或者 App 內不支援 Codespaces 的完整功能，流程會卡住。
- **痛點 2**：手機瀏覽器上的 VS Code 介面非常擁擠，對於無經驗用戶來說，找到左側的檔案總管並新建檔案是一大挑戰。

### 步驟 3：使用 Claude.ai 執行 Prompt
- **預期**：用戶複製 `ultimate_one_prompt.md` 並貼入 Claude.ai。
- **實際體驗**：Claude.ai 完美理解了 Prompt，並開始生成代碼。
- **痛點 3**：對於「AI 圖片生成平台」，代碼量非常大（至少 15+ 個檔案）。Claude.ai 受到輸出長度限制，可能會在生成一半時停住，需要用戶輸入「繼續」。

### 步驟 4：搬運代碼 (最痛苦的環節)
- **預期**：用戶將 Claude 生成的代碼複製到 Codespaces 中。
- **實際體驗**：這是整個流程中最脆弱的一環。
- **痛點 4**：手機上長按選擇大量代碼非常困難，容易漏選或多選。
- **痛點 5**：用戶需要在 Codespaces 中手動建立 15+ 個檔案與目錄（例如 `app/api/generate/route.ts`），這對於無經驗用戶來說極易出錯。只要檔名錯一個字，整個專案就無法運行。

### 步驟 5：同步與部署
- **預期**：用戶點擊 `Sync Changes`，然後去 Zeabur 部署。
- **實際體驗**：同步成功，Zeabur 也成功觸發部署。
- **痛點 6**：如果代碼有報錯（例如缺少依賴或 Prisma schema 錯誤），Zeabur 部署會失敗。無經驗用戶看到一堆紅色的錯誤日誌會不知所措，不知道如何將錯誤反饋給 Claude 修復。

---

## 2. 深度缺陷分析

基於上述模擬，我總結出本框架目前的 **3 大致命缺陷**：

### 缺陷一：手動搬運代碼的「人肉編譯器」困境
要求無代碼經驗的用戶在手機上手動創建 15+ 個檔案並準確貼上代碼，是不切實際的。這不僅效率低下，而且出錯率高達 90% 以上。

### 缺陷二：Claude 的上下文長度與輸出截斷
一個商業級專案的初始代碼量通常在 500-1000 行以上。Claude.ai 網頁版單次回答的長度限制會導致代碼被截斷，破壞了「一鍵完成」的體驗。

### 缺陷三：缺乏自動化的錯誤修復機制 (Broken Loop)
當 Zeabur 部署失敗時，目前的流程需要用戶手動複製錯誤日誌，貼回 Claude，再把 Claude 的修復代碼貼回 Codespaces。這打破了 Loop Engineering 的「自動閉環」。

---

## 3. 改進方案設計

為了徹底解決這些缺陷，我們需要從「依賴用戶手動操作」轉向「完全由機器腳本代勞」。

### 解決方案 A：引入「一鍵下載腳本」(解決缺陷一)
我們不能讓用戶手動創建檔案。我們需要讓 Claude 輸出一個**單一的 Bash 腳本**（或 Node.js 腳本），這個腳本包含了創建所有目錄和檔案的命令。用戶只需要在 Codespaces 終端機貼上這**一行代碼**，所有檔案就會自動生成。

### 解決方案 B：分階段微型架構 (解決缺陷二)
我們不能指望 Claude 一次性輸出所有代碼。我們應該在 Prompt 中要求 Claude 先輸出一個「最小可行性產品 (MVP)」，確保能跑通部署流程。後續再透過追加 Prompt 逐步添加功能。

### 解決方案 C：整合 GitHub Actions 自動化測試 (解決缺陷三)
我們需要在專案初始化時，自動加入 GitHub Actions 工作流。當用戶推送代碼時，GitHub Actions 會先進行構建測試。如果失敗，Actions 會生成明確的錯誤報告，用戶只需將這個報告貼給 Claude 即可。
