# 互動式 API Key 詢問指令

這份 prompt 用在「技術棧已大致確定，但外部服務憑證仍未齊」的情境。

## 目標

- 只問必要的 key
- 不阻斷主流程
- 缺 key 時提供最短官方取得方式與 fallback

## 使用規則

1. 先判斷目前 task 是否真的需要外部服務
2. 若不需要，跳過這一段
3. 若需要，只問 1-2 個服務
4. 允許使用者：
   - 已有 key
   - 先跳過
   - 沿用現有環境變數
   - 請 AI 解釋用途

## 建議問法

```text
我發現目前方案可能需要以下外部服務。請告訴我您要怎麼處理：

1. 我已有 [Service Name] 的 key / 連線資訊
2. 先跳過，請用 placeholder 繼續
3. 沿用現有專案環境變數
4. 先解釋這個服務是做什麼的

您可以直接回覆代號，也可以直接提供必要資訊。
```

## AI 的後續行為

### 使用者已有 key

- 收下必要資訊
- 不在公開文件中明文重貼
- 只把需要的欄位寫進 env 模板或設定指引

### 使用者先跳過

- 產生 placeholder
- 在 `docs/STATE.md` 記錄缺口
- 若有 fallback，就啟用 fallback

### 使用者沿用現有設定

- 先檢查 repo 是否已有對應 env 名稱
- 沒把握時，先確認名稱與用途

### 使用者要解釋

- 用 1-3 句說明：
  - 這個服務做什麼
  - 為什麼現在需要
  - 若不使用的替代方案

## 安全規則

- 不要求使用者把敏感資訊提交到 git
- 不要求把 key 寫進 README
- 所有秘密資料都走環境變數或平台 secret 設定

## 低 Token 規則

- 先短答，再展開
- 已確認的 key 不重問
- 能 placeholder 就先 placeholder
- 沒有 key，不代表流程停住
